# Flutter Todo App

A Flutter mobile application that interfaces with the [JSONPlaceholder API](https://jsonplaceholder.typicode.com/) to manage a task list. Demonstrates the BLoC pattern, RESTful API integration, offline support with local caching, and optimistic UI updates.

---

## Setup Instructions

### Prerequisites

- Flutter SDK `>=3.11.5`
- Dart SDK `>=3.11.5`
- An iOS simulator, Android emulator, or physical device

### Steps

1. **Clone / open the project**

2. **Install dependencies**

   ```bash
   flutter pub get
   ```

3. **Generate DI code** (required before first run)

   ```bash
   dart run build_runner build
   ```

4. **Run the app**

   ```bash
   flutter run
   ```

## BLoC Pattern Implementation

### AuthBloc

`AuthBloc` extends `HydratedBloc` so the authenticated session persists across app restarts without requiring the user to log in again.

**Events:** `LoginRequested`, `LogoutRequested`

**States:** `AuthInitial`, `AuthLoading`, `AuthAuthenticated`, `AuthFailure`

`fromJson`/`toJson` serialize only `AuthAuthenticated` (storing `username`). All other states restore as `AuthInitial`.

### TodoBloc

`TodoBloc` extends `HydratedBloc` to persist the todo list across cold starts (shows cached data instantly before the network call completes).

**Events:**

| Event | Description |
|---|---|
| `LoadTasks` | Load from cache then fetch remote (uses `droppable()` transformer) |
| `RefreshTasks` | Force network fetch (triggered by pull-to-refresh) |
| `AddTask(title)` | Optimistic add — prepends a temp entity immediately |
| `CompleteTask(id)` | Optimistic complete — flips `completed` immediately |
| `DeleteTask(id)` | Optimistic delete — removes immediately |
| `SearchTasks(query)` | Client-side filter, debounced 300ms in the UI |
| `ConnectivityChanged(isOnline)` | Triggered by connectivity stream; starts sync on reconnect |
| `SyncPendingOperations` | Processes the offline sync queue |

**State** is a single `TodoState` class with `copyWith`, containing both the full list and filtered list. This avoids the complexity of mapping between sealed state subclasses during optimistic updates.

## Offline Support Strategy

### 1. Hive Local Cache

All todos fetched from the API are stored in a Hive box (`todos_box`). On subsequent launches or when offline, the app immediately renders cached data. The `TodoBloc.fromJson`/`toJson` also persists the todo list via `HydratedBloc`, giving a two-layer cache: HydratedBloc (fast, in-memory-backed) and Hive (persistent).

### 2. Optimistic Updates

When the user adds, completes, or deletes a todo, the UI updates **immediately** before the network call completes:

- **Add**: A temporary entity with a negative ID is prepended to the list.
- **Complete**: The `completed` flag is flipped in place.
- **Delete**: The entity is removed from the list.

If the network call fails, the change is **rolled back** to the previous state and an error `SnackBar` is shown.

### 3. Offline Sync Queue

When a write operation (add/complete/delete) is attempted offline, the operation is serialized to a Hive box (`sync_queue_box`) as a map:

```
{'op': 'add', 'title': 'Buy groceries', 'tempId': -1716123456789}
{'op': 'complete', 'id': 42}
{'op': 'delete', 'id': 7}
```

### 4. Sync on Reconnect

`ConnectivityService` wraps `connectivity_plus` and exposes a `Stream<bool>`. `TodoBloc` subscribes to this stream. When the connection transitions from offline to online, `SyncPendingOperations` is dispatched automatically, which processes the queue sequentially and then refreshes the todo list.

## Design Decisions

**Single TodoState vs sealed states**: A single state class with `copyWith` was chosen over a sealed hierarchy because multiple fields change simultaneously during optimistic updates. A sealed hierarchy would require complex state mapping.

**Repository as DI factory, Blocs as widget-scoped**: `TodoBloc` is created via `BlocProvider` in `TodoListScreen` using `getIt` to inject dependencies. This keeps BLoC lifecycle tied to the screen and avoids stale state if the screen is re-entered. `AuthBloc` is a DI singleton because it must be accessible by both the widget tree and the GoRouter redirect guard.

**Manual Hive adapter**: `hive_generator` (code gen for Hive adapters) conflicts with `injectable_generator` on the `source_gen` version. The `TodoModelAdapter` was written manually instead — it is straightforward for a 5-field model and avoids the dependency conflict entirely.

**GoRouter auth redirect**: `GoRouterRefreshStream` wraps `AuthBloc.stream` as a `ChangeNotifier` so GoRouter re-evaluates the `redirect` callback on every auth state change.

## Challenges and Solutions

| Challenge | Solution |
|---|---|
| `hive_generator` ↔ `injectable_generator` conflict on `source_gen` version | Dropped `hive_generator`, wrote the `TodoModelAdapter` manually |
| `bloc_concurrency 0.2.x` requires `bloc ^8`, incompatible with `hydrated_bloc ^11` (which needs `bloc ^9`) | Upgraded to `bloc_concurrency ^0.3.0` |
| `HydratedStorage.build` in v11 requires `HydratedStorageDirectory` not `Directory` | Wrapped the path: `HydratedStorageDirectory(dir.path)` |
| Dart private classes (`_ClassName`) are file-scoped, preventing `_ConnectivityChanged` from being referenced in `todo_bloc.dart` | Renamed the internal events to `ConnectivityChanged` / `SyncPendingOperations` (non-private) |
| `connectivity_plus 4.0.2` returns `ConnectivityResult` (not `List<ConnectivityResult>`) | Compared with `!= ConnectivityResult.none` directly |
